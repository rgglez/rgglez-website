## Introducción

En 2018, el equipo de Go presentó los "módulos" como el nuevo estándar
para manejar dependencias en proyectos Go. Un módulo agrupa paquetes
relacionados que se versionan en conjunto e incluye lo necesario para
compilar una base de código. Desde Go 1.11, con Go Modules, esta gestión
se volvió más simple, flexible y oficialmente respaldada por el proyecto.
Entre sus principales capacidades están:

- Versionado de dependencias usando Versionado Semántico (SemVer).
- Comandos de gestión de dependencias simplificados (por ejemplo, `go get`,
  `go mod tidy`).
- Generación automática de un archivo de manifiesto (go.mod) con información
  detallada sobre las dependencias.
- Descarga automática y almacenamiento en caché de las dependencias necesarias.

Con Go Modules ya no hace falta ubicar el código dentro de `$GOPATH`,
restricción que antes condicionaba el desarrollo en Go. Este enfoque ofrece
una estructura de proyecto más flexible y facilita el trabajo al alternar
entre distintos repositorios.

Los módulos de Go ofrecen una forma potente de reutilizar componentes propios y
de terceros en nuestros programas. La librería estándar de Go incluye una gran
cantidad de paquetes, pero en muchos casos se necesitan módulos de otros autores
para funciones específicas, o módulos desarrollados dentro de la organización
que, al tratarse de software propietario, no queremos publicar.

Esto plantea un problema: un módulo alojado en un repositorio
público, por ejemplo en GitHub, se puede instalar fácilmente con:

```
go get github.com/gofiber/fiber/v3
```

Flujo normal de `go get` (módulo público en GitHub):

```text
┌──────────────┐
│ Desarrollador│
│  go get ...  │
└──────┬───────┘
       │
       v
┌──────────────┐
│  Comando Go  │
│ (resolver mod│
│ y versión)   │
└──────┬───────┘
       │ consulta
       v
┌─────────────────────────┐
│ proxy.golang.org        │
│ (zip/mod/info cacheados)│
└──────┬──────────────────┘
       │ verifica checksums
       v
┌──────────────────────────┐
│ sum.golang.org           │
│ (transparencia e integr.)│
└──────┬───────────────────┘
       │ descarga módulo
       v
┌─────────────────────────┐
│ Cache local de módulos  │
│ $GOMODCACHE             │
└──────┬──────────────────┘
       │ actualiza
       v
┌─────────────────────────┐
│ go.mod / go.sum         │
└─────────────────────────┘
```

Pero, un módulo ubicado en un repositorio privado (p. ej. de GitHub) suele fallar
al ejecutar `go get`. Esto ocurre porque Go intenta resolver dependencias con el
proxy y la base de checksums públicos por defecto, y además Git (utilizado
internamente por `go get`) necesita credenciales válidas para clonar el
repositorio privado. Si no configuras autenticación (HTTPS con token o SSH)
y variables como `GOPRIVATE`, la descarga termina en errores de acceso.

## Contexto

| Elemento | Valor |
|---|---|
| Repositorio | `https://github.com/myorg/my-go-module` (privado) |
| Consumidores | Programas Go internos de **myorg** |
| SO de desarrollo | Debian 13 (Trixie) |

Los desarrolladores ya pertenecen a un **team** de la organización `myorg` con
permiso **write** sobre el repositorio. La guía asume que cada desarrollador
ejecuta los pasos en su propia máquina.

---

## ¿Qué flujo me conviene?

| | HTTPS + Token | SSH (llave personal) | SSH (deploy key) | Híbrido (deploy key + HTTPS) |
|---|---|---|---|---|
| Autenticación | Personal Access Token (PAT) | Par de llaves del usuario | Par de llaves por repositorio | Deploy key para leer, PAT para escribir |
| Alcance | Todos los repos del usuario | Todos los repos del usuario | Un solo repositorio | Lectura: un repo; escritura: todos los del PAT |
| Revocación centralizada | El admin revoca tokens desde la consola de GitHub | El admin remueve al usuario del team | El admin elimina la deploy key del repo | Combina ambos mecanismos |
| Expiración | Obligatoria en fine-grained, configurable en classic | Las llaves no expiran por defecto | Las llaves no expiran por defecto | Según tipo de token y llave |
| Firewalls corporativos | Funciona siempre (puerto 443) | Puede fallar si el puerto 22 está bloqueado | Igual que SSH personal | Lectura: puerto 22; escritura: puerto 443 |
| Credenciales por host | Una sola por `github.com`; el token debe cubrir todos los repos | Sin conflicto | Requiere alias en `~/.ssh/config` por repo | Requiere alias + credential helper |
| Secreto en disco | Token en `~/.git-credentials` (texto plano) | Llave privada en `~/.ssh/` (con passphrase) | Igual que SSH personal | Ambos: llave + token |
| CI/CD | Más simple: un string en un secret | Requiere montar archivo de llave | Ideal: acceso mínimo a un solo repo | Poco común en CI/CD |
| Quién configura | Cada desarrollador crea su token | Cada desarrollador crea su llave | El admin agrega la llave pública; el dev genera el par | El admin agrega deploy key; el dev configura ambos canales |

> **Fine-grained vs Classic token**: un fine-grained token permite seleccionar
> repositorios específicos y tiene expiración obligatoria; un classic token con
> scope `repo` cubre todos los repos del usuario y la expiración es opcional.

**Elige un flujo y sigue solo esa sección.** Cada una es autocontenida.

# Flujo A — HTTPS con Personal Access Token

---

## A1. Paquetes

```bash
sudo apt update
sudo apt install -y git golang-go
```

Verifica:

```bash
git --version   # ≥ 2.x
go version      # ≥ 1.21
```

---

## A2. Personal Access Token (PAT)

Como `~/.git-credentials` almacena **una sola credencial por host**, el token
que uses para `github.com` debe cubrir **todos** los repositorios con los que
trabajes (públicos y privados), no solo `my-go-module`. De lo contrario, Git
presentará el token restringido para cualquier otro repo y obtendrás 403.

> **Importante**: el token no amplía los permisos del usuario. Solo puede
> acceder a repositorios que el usuario ya tiene permitidos por su membresía
> en teams. El administrador de la organización sigue controlando el acceso
> de forma centralizada.

### Si ya tienes un Classic PAT con scope `repo`

**No necesitas crear otro token.** Ese token ya cubre todos los repositorios
a los que tu usuario tiene acceso. Pasa directamente al paso A3.

Verifica en **Settings → Developer settings → Personal access tokens →
Tokens (classic)** que el scope `repo` esté marcado.

### Si necesitas crear un token nuevo

Elige **una** variante:

**Fine-grained token (recomendado)**

1. Ve a **Settings → Developer settings → Personal access tokens →
   Fine-grained tokens → Generate new token**.
2. Configura:
   - **Token name**: algo descriptivo, p.ej. `dev-myorg`.
   - **Expiration**: según la política de tu organización.
   - **Resource owner**: selecciona **myorg**.
   - **Repository access**: selecciona **All repositories**.
   - **Permissions → Repository permissions**:
     - **Contents**: `Read and write`.
     - **Metadata**: `Read-only` (se habilita automáticamente).
3. Haz clic en **Generate token** y **copia el token** (no lo volverás a ver).

**Classic token**

1. Ve a **Settings → Developer settings → Personal access tokens →
   Tokens (classic) → Generate new token (classic)**.
2. Configura:
   - **Note**: algo descriptivo, p.ej. `dev-myorg`.
   - **Expiration**: según la política de tu organización.
   - **Scopes**: marca **`repo`**.
3. Haz clic en **Generate token** y **copia el token**.

## A3. Credential Helper

El objetivo es que tanto `git` como `go get` usen el token de forma
transparente, sin que te lo pida cada vez. Elige **una** opción:

**Opción 1 — `git-credential-store` (archivo en disco, más simple)**

```bash
git config --global credential.helper store
```

Si ya existía `~/.git-credentials` con una entrada para `github.com`, hay que
reemplazarla (Git usa la primera coincidencia):

```bash
sed -i '\|://.*@github\.com|d' ~/.git-credentials 2>/dev/null
```

Almacena la credencial:

```bash
# Sustituye TU_USUARIO y TU_TOKEN
printf 'protocol=https\nhost=github.com\nusername=TU_USUARIO\npassword=TU_TOKEN\n' \
  | git credential approve
```

Verifica y protege:

```bash
cat ~/.git-credentials
# Deberías ver: https://TU_USUARIO:TU_TOKEN@github.com
chmod 600 ~/.git-credentials
```

**Opción 2 — `git-credential-cache` (en memoria, más seguro)**

```bash
git config --global credential.helper 'cache --timeout=28800'
```

Recordará las credenciales durante 8 horas. Tendrás que reintroducirlas al
expirar o reiniciar.

**Opción 3 — `gh auth` (GitHub CLI)**

```bash
sudo apt install -y gh
gh auth login          # Elige HTTPS, pega tu PAT
gh auth setup-git      # Registra gh como credential helper
```

---

## A4. Forzar HTTPS para Go

Go puede intentar usar SSH al resolver módulos. Para que siempre use HTTPS:

```bash
git config --global url."https://github.com/".insteadOf "ssh://git@github.com/"
git config --global url."https://github.com/".insteadOf "git@github.com:"
```

Verifica:

```bash
git config --global --get-regexp url
# url.https://github.com/.insteadof ssh://git@github.com/
# url.https://github.com/.insteadof git@github.com:
```

## A5. Variables de Go para módulos privados

Go intenta validar módulos contra el proxy público y la base de checksums.
Ambos fallan con repos privados. Lista **solo** los módulos Go privados
que usas como dependencias, no todos los repos de la org:

```bash
cat >> ~/.bashrc << 'EOF'

# --- Go: módulos privados de myorg ---
export GOPRIVATE="github.com/myorg/my-go-module"
export GONOSUMDB="github.com/myorg/my-go-module"
export GONOSUMCHECK="github.com/myorg/my-go-module"
EOF

source ~/.bashrc
```

| Variable | Efecto |
|---|---|
| `GOPRIVATE` | Indica a Go que los módulos listados son privados. Implica `GONOSUMDB` y `GONOPROXY`. |
| `GONOSUMDB` | No consulta `sum.golang.org` para estos módulos. |
| `GONOSUMCHECK` | No verifica checksums contra la base de datos pública. |

Verifica:

```bash
go env GOPRIVATE GONOSUMDB GONOSUMCHECK
# Las tres deben mostrar github.com/myorg/my-go-module
```

> **Varios módulos privados**: sepáralos con comas, sin espacios:
> ```
> export GOPRIVATE="github.com/myorg/my-go-module,github.com/myorg/otro-modulo"
> ```
> Repite el mismo patrón en `GONOSUMDB` y `GONOSUMCHECK`.

## A10. Solución de problemas

| Síntoma | Causa probable | Solución |
|---|---|---|
| `410 Gone` al hacer `go get` | Go intenta usar `proxy.golang.org` | Verifica que `GOPRIVATE` incluye `github.com/myorg/my-go-module` |
| `SECURITY ERROR` de checksums | `GONOSUMCHECK` no está definido | Agrega `export GONOSUMCHECK="github.com/myorg/my-go-module"` |
| `go get` descarga versión vieja | Caché local de Go | Ejecuta `go clean -modcache` y reintenta |
| `fatal: could not read Username` | Credential helper no configurado | Ejecuta el paso A3 |
| `403 Forbidden` | Token sin permisos suficientes | Verifica que el PAT tiene permiso **Contents: Read and write** |
| `403` en repos que antes funcionaban | Token fine-grained restringido a repos específicos | Recrea con **All repositories** o usa Classic PAT con scope `repo` |
| Go intenta conectar por SSH | Falta la regla `insteadOf` | Ejecuta el paso A4 |

# Flujo B — SSH con llave personal

---

## B1. Paquetes

```bash
sudo apt update
sudo apt install -y git golang-go openssh-client
```

Verifica:

```bash
git --version   # ≥ 2.x
go version      # ≥ 1.21
ssh -V          # OpenSSH ≥ 9.x
```

---

## B2. Generar la llave SSH

Si ya tienes un par de llaves en `~/.ssh/id_ed25519`, puedes reutilizarlo.
Si no:

```bash
ssh-keygen -t ed25519 -C "tu-email@ejemplo.com"
```

Acepta la ruta por defecto y **establece un passphrase**.

Verifica:

```bash
ls -l ~/.ssh/id_ed25519 ~/.ssh/id_ed25519.pub
```

---

## B3. Agregar la llave pública a GitHub

Copia la llave pública:

```bash
cat ~/.ssh/id_ed25519.pub
```

En GitHub:

1. Ve a **Settings → SSH and GPG keys → New SSH key**.
2. Configura:
   - **Title**: p.ej. `debian-dev-myorg`.
   - **Key type**: `Authentication Key`.
   - **Key**: pega el contenido.
3. Haz clic en **Add SSH key**.

---

## B4. Configurar `ssh-agent`

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
```

Para auto-inicio en cada sesión:

```bash
cat >> ~/.bashrc << 'AGENT'

# --- SSH agent automático ---
if [ -z "$SSH_AUTH_SOCK" ]; then
  eval "$(ssh-agent -s)" > /dev/null
  ssh-add ~/.ssh/id_ed25519 2>/dev/null
fi
AGENT

source ~/.bashrc
```

> **Alternativa**: en GNOME o KDE, el keyring del sistema suele actuar como
> agente SSH automáticamente.

## C2. Generar un par de llaves dedicado

Usa un nombre de archivo que identifique el repo:

```bash

  -f ~/.ssh/deploy_my-go-module
```

Establece un passphrase si lo deseas (recomendado en máquinas de desarrollo;
en CI/CD normalmente se deja vacío).

> **Restricción de GitHub**: una misma llave pública no puede ser deploy key
> de más de un repositorio. Si necesitas deploy keys para varios repos,
> genera un par distinto para cada uno.

---

## C3. Registrar la llave pública en el repositorio

Copia la llave pública:

```bash
cat ~/.ssh/deploy_my-go-module.pub
```

En GitHub:

1. Ve al repositorio **myorg/my-go-module → Settings → Deploy keys →
   Add deploy key**.
2. Configura:
   - **Title**: p.ej. `dev-rodolfo` o `ci-build-server`.
   - **Key**: pega el contenido.
   - **Allow write access**: márcalo si necesitas hacer push.
     Déjalo desmarcado si solo vas a consumir el módulo con `go get`.
3. Haz clic en **Add key**.

> **Nota para el admin**: solo usuarios con permiso **admin** sobre el
> repositorio pueden agregar deploy keys. Puedes agregar, ver y eliminar
> deploy keys desde la configuración del repo.

---

## C4. Alias en `~/.ssh/config`

Con una deploy key, SSH necesita saber **cuál llave usar para cuál repo**.
Esto se resuelve con un alias de host:

```bash
cat >> ~/.ssh/config << 'SSHCFG'

# --- Deploy key: myorg/my-go-module ---
Host github-my-go-module
  HostName github.com
  User git
  IdentityFile ~/.ssh/deploy_my-go-module
  IdentitiesOnly yes
SSHCFG

chmod 600 ~/.ssh/config
```

| Directiva | Función |
|---|---|
| `Host github-my-go-module` | Alias arbitrario que usarás en URLs de Git y en `insteadOf`. |
| `HostName github.com` | El servidor real al que SSH se conectará. |
| `User git` | GitHub siempre usa el usuario `git` para SSH. |
| `IdentityFile` | La llave privada de la deploy key. |
| `IdentitiesOnly yes` | Impide que `ssh-agent` ofrezca otras llaves. |

> **Múltiples deploy keys**: agrega un bloque `Host` por cada repo con un
> alias distinto y su propio `IdentityFile`.

## D6. Personal Access Token para escritura

Los pushes irán por HTTPS, así que necesitas un PAT configurado en un
credential helper. Si ya tienes uno funcionando, puedes reutilizarlo.
Si no:

### Crear el token

Sigue las instrucciones del paso A2 (Classic con scope `repo` o fine-grained
con **All repositories** y **Contents: Read and write**).

### Configurar el credential helper

```bash
git config --global credential.helper store

# Limpiar entrada previa de github.com
sed -i '\|://.*@github\.com|d' ~/.git-credentials 2>/dev/null

# Almacenar la credencial
printf 'protocol=https\nhost=github.com\nusername=TU_USUARIO\npassword=TU_TOKEN\n' \
  | git credential approve

chmod 600 ~/.git-credentials
```

## D7. Reglas `insteadOf` + `pushInsteadOf`

Aquí está la pieza clave. Git aplica `insteadOf` primero, y luego
`pushInsteadOf` sobreescribe solo para operaciones de escritura:

```bash
# Lecturas (go get, git pull, git clone) → SSH deploy key
git config --global url."git@github-my-go-module:myorg/my-go-module".insteadOf \
  "https://github.com/myorg/my-go-module"

# Pushes → de vuelta a HTTPS (usa el PAT del credential helper)
git config --global url."https://github.com/myorg/my-go-module".pushInsteadOf \
  "git@github-my-go-module:myorg/my-go-module"
```

El flujo resultante:

| Operación | Cadena de reescritura | Autenticación |
|---|---|---|
| `go get` | HTTPS → SSH (deploy key) | Deploy key, solo lectura |
| `git pull` | HTTPS → SSH (deploy key) | Deploy key, solo lectura |
| `git clone` | HTTPS → SSH (deploy key) | Deploy key, solo lectura |
| `git push` | HTTPS → SSH → HTTPS | PAT, lectura/escritura |

El desarrollador usa URLs HTTPS normales para todo, y Git resuelve por debajo
qué credencial usar según la dirección de la operación.

Verifica:

```bash
git config --global --get-regexp url
# url.git@github-my-go-module:myorg/my-go-module.insteadof https://github.com/myorg/my-go-module
# url.https://github.com/myorg/my-go-module.pushinsteadof git@github-my-go-module:myorg/my-go-module
```
